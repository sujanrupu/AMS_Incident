Memory Leak Issue
Ticket 1 — Fresh RCA
Summary: Application memory continuously increasing — OOM kills on ECOM-APP pod
Description: The e-commerce application pod ECOM-APP is experiencing a continuous memory leak. Memory starts at 512MB after restart and grows by 50MB per hour without releasing. After 12 hours the pod hits the 1GB memory limit and gets OOM killed by Kubernetes. A new caching layer introduced last sprint is suspected of not evicting entries correctly. The OOM kill cycle repeats every 12 hours impacting production stability.
Ticket 2 — Above 0.60, same issue → LLM confirms → copy RCA
Summary: Memory growing without release — pod OOM killed repeatedly every few hours
Description: ECOM-APP Kubernetes pod memory usage grows continuously from 500MB after each restart hitting the 1GB memory limit every 8-10 hours triggering OOM kills. Garbage collection is not releasing memory between requests. A cache component introduced in the recent sprint accumulates entries without eviction causing the steady growth. Pod restarts provide only temporary relief before the memory leak cycle begins again immediately after startup.
Ticket 3 — Below 0.60, different issue → fresh generation
Summary: PostgreSQL shared_buffers misconfiguration causing excessive RAM consumption on DB server
Description: The PostgreSQL database server is consuming 90% of system RAM due to shared_buffers being set to 80% of total memory in postgresql.conf which is far above the recommended 25%. This is not an application memory leak but a database server misconfiguration. The OS is swapping heavily causing database query performance to degrade. The application pods are healthy and memory usage is normal. Only the database host is affected by the RAM exhaustion.
